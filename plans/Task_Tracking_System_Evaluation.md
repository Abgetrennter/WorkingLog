# 未完成日志条目的任务追踪系统 - 设计与代码实现评估报告

## 1. 整体架构与机制概述

该项目针对“未完成日志条目（跨日任务）”实现了一套基于 **每日快照 (Daily Snapshot)** 结合 **追踪标识 (TrackingId)** 的任务追踪系统。由于底层数据源是按月/按日划分的 Excel/JSON 数据结构，系统并没有一个全局单一的“任务表”，而是通过在每天的日志中复制未完成的条目来实现跨日流转。

核心机制包括：
1. **标识关联**：通过 `WorkLogItem.TrackingId` 关联分布在不同日期的同一任务。
2. **状态驱动 UI**：根据 `StatusEnum`（待办 Todo、进行中 Doing）动态切换表单界面，提供“当日进展”填报专属 UI。
3. **跨日自动结转**：在 `MainForm.cs` 中，每天首次加载时，自动从上一工作日拉取状态为“进行中 (Doing)”的条目。
4. **全生命周期追溯**：在任务标记为“已完成 (Done)”时，依靠 `TrackingId` 往回追溯，收集过去多天的进展并生成汇总文本。

## 2. 详细设计与实现评估

### 2.1 数据模型与状态定义 (模型层)
*   **设计合理性：高**
    *   引入了 `StatusEnum` (Todo, Doing, Done, Blocked, Cancelled)，清晰定义了任务生命周期。
    *   在 `WorkLogItem` 中引入了全局唯一的 `TrackingId`。这是一个非常关键且优秀的设计，巧妙地解决了无关系型数据库、按日归档数据结构下的任务级联问题。
    *   **改进空间**：`TodoForm` (便签式待办) 完全脱离了 `WorkLogItem` 体系，仅向本地写入 `todo.txt`，导致系统内存在两套“待办”概念，极易引起用户混淆。

### 2.2 跨日结转逻辑 (核心业务层)
*   **代码位置**：`MainForm.CheckAndInheritItems()`
*   **完成度：中等 (存在明显逻辑漏洞)**
*   **优点**：
    *   智能跳过周末（寻找上一个工作日），符合真实工作场景。
    *   在结转时，使用 Regex 清理了上一日的“当日进展”文本块，保证新一天的任务内容干净整洁，仅保留核心描述。
*   **缺陷**：
    *   **严重漏洞**：代码 `var inherited = prevItems.Where(i => i.Status == StatusEnum.Doing).ToList();` **仅仅结转了“进行中(Doing)”的任务**。状态为“待办(Todo)”或“阻塞(Blocked)”的任务会被遗留在了过去，永远不会被自动带到今天。这违背了任务追踪的基本诉求（未完成均应结转）。

### 2.3 进度汇报与完结汇总 (UI/交互层)
*   **代码位置**：`ItemEditForm.cs` (`OnAddProgressClick`, `OnCompleteClick`, `TraceBackAndMergeProgress`)
*   **完成度：高 (体验好，但存在数据结构隐患)**
*   **优点**：
    *   表单根据状态自动隐藏“开始/结束时间”，改为显示“当日进展”输入框，用户体验针对性强。
    *   `TraceBackAndMergeProgress` 是一个非常惊艳的功能：在点击“完成”时，系统会自动往前回溯，依靠 `TrackingId` 抓取该任务过去每一天的“当日进展”，最终生成 `【项目全周期进展汇总】` 并合并到最终的日志内容中。
*   **隐患**：
    *   **脆弱的字符串依赖**：当日进展是以特殊字符串格式（如 `——————————\n【yyyy-MM-dd 进展】\n...`）直接 Hardcode 拼接到 `ItemContent` 正文中的。读取和清理高度依赖正则表达式（Regex）。一旦用户在正文中手动修改破坏了格式，追溯和结转清理功能就会立刻失效。
    *   **追溯中断机制**：追溯逻辑中设置了 `notFoundCount <= 3` 即如果在连续 3 天的日志中没找到该 `TrackingId` 就停止追溯。如果某个任务被搁置了 4 天后再次启动并完成，更早之前的进度就会丢失。

## 3. 总体评价与重构建议

**总体评价**：
该任务追踪系统在**概念设计上非常优秀**，特别是利用 `TrackingId` 将每日日志流水串联成完整任务生命周期的思路，非常契合目前免数据库的 Excel/JSON 持久化架构。但在**代码实现细节和健壮性上存在瑕疵**。

### 核心改进建议 (Action Items)：

1. **修复结转漏单 Bug (最高优)**
   修改 `MainForm.cs` 中的继承过滤条件，应将 `Todo`、`Doing`、`Blocked` (即 `StatusHelper.IsUnfinished()`) 统统纳入自动结转范围，而不仅仅是 `Doing`。
2. **结构化“当日进展”数据 (架构优化)**
   停止将进展内容直接污染到 `ItemContent` 中。建议在 `WorkLogItem` 模型中新增一个字段（如 `public string DailyProgress { get; set; }` 或使用专门的集合），并在 UI 展示时再进行拼装。这样能彻底摆脱对容易出错的 Regex 字符串匹配的依赖。
3. **统一 Todo 概念 (产品层面)**
   废弃或重新定位独立且无关联的 `TodoForm` (todo.txt)。建议将该快速便签功能更名为“草稿本(Draft)”，而真正的“待办”应全部纳入 `WorkLogItem` 的标准数据流转中。
4. **优化追溯中断逻辑**
   将连续 3 天未找到即停止的硬编码逻辑，改为可配置或通过维持一个全局 `TrackingId` 索引表（内存字典）来精确定位任务历史，避免因长期搁置导致的总结丢失。
